# Graphiti 阅读攻略

项目地址：
- 相关方法论文：[Zep: A Temporal Knowledge Graph Architecture for Agent Memory](https://arxiv.org/abs/2501.13956)
- GitHub：[getzep/graphiti](https://github.com/getzep/graphiti)

如果你想研究“图记忆不是概念，而是一个真正的工程系统怎么做”，`Graphiti` 是目前最值得啃的公开项目之一。  
它和 `Mem0`、`Letta` 的最大区别在于：它不是把 memory 看成摘要条目池，而是把 memory 看成**带时间语义的知识图引擎**。

这意味着你读它时，脑子里要换一套问题：

- 实体怎么抽出来？
- 关系边怎么存？
- 时间有效期怎么表示？
- 图更新和图查询怎么做？
- 原始 episode 与图内事实怎么绑定？

## 一、先用一句话理解 Graphiti

`Graphiti` 可以理解成：**面向 agent memory 的 temporal knowledge graph engine。**

它最重要的价值，不在于“能存图”这件事本身，而在于它把很多在普通向量 memory 里被压扁的关系显式化了，例如：

- 谁和谁相关；
- 哪个事件依赖哪个事件；
- 哪个事实在哪个时间范围内有效；
- 某条关系来自哪次交互。

## 二、打开仓库后先看什么

建议第一轮重点看：

- `graphiti_core/`
- `server/`
- `mcp_server/`
- `tests/`
- `spec/`

### 1. `graphiti_core/`：真正的图记忆内核

这里是整个项目的心脏。  
如果你只读一个目录，那就读这个。

你在这一层最该理解的是：

- episode / entity / edge / claim / summary 这些对象是什么关系；
- 时间有效性是怎么被表示的；
- 写入一段新交互时，图内对象是如何被更新的；
- retrieval 时，是按节点、按边、按时间还是按 subgraph 来拿候选。

这部分和普通 memory repo 完全不一样，因为它不只是存文本，而是在维护一个可演化的事实图。

### 2. `server/`：把图内核暴露成服务

这层值得看，因为它说明项目不满足于“本地研究代码”。  
它在认真回答：

- 图记忆服务如何对外提供 API；
- 上层 agent 如何请求图 memory；
- 服务边界与图内核如何拆分。

对你以后自己做系统特别有帮助的一点是：  
**图内核不一定要和 agent runtime 写死在一起。**

### 3. `mcp_server/`：和 agent 工具生态的连接点

这部分很加分。  
它说明 `Graphiti` 的目标不只是做数据库，而是要作为 agent 可调用的能力暴露出去。  
你从这一层能学到：

- 图记忆怎样以工具形式被上层系统使用；
- memory engine 和 agent client 之间的接口思维；
- 为什么“图记忆引擎”要有面向 agent 的薄交互层。

### 4. `tests/`：这是判断图系统有没有工程感的关键

图系统很容易看起来很强，但难点在于它是否稳定。  
你看 tests 时应该关心：

- 节点去重是不是有测试；
- 时间有效性是不是有测试；
- 子图检索是不是有测试；
- 新 episode 写入后图结构是否按预期演化。

### 5. `spec/`：这是结构化项目的信号

如果一个图记忆项目带有 `spec/` 这样的位置，通常说明它对接口和行为边界是有意识的。  
这类目录很适合用来建立整体心智模型，因为它往往更接近“作者希望系统如何工作”，而不是“当前某个实现细节恰好这样写了”。

## 三、一个高效的阅读顺序

建议你这么读：

1. 根 README，看项目自我定位
2. `graphiti_core/`，建立对象模型
3. `tests/`，理解核心行为保证
4. `server/`，理解服务边界
5. `mcp_server/`，理解 agent 集成方式
6. `spec/`，最后回头做抽象整理

顺序上一定不要先钻 API 层。  
因为 Graphiti 的价值不在“有没有一个接口叫 create_memory”，而在于它背后维护的是一套怎样的 temporal graph model。

## 四、读 Graphiti 时最值得学的 5 件事

### 1. 关系是第一公民，不是检索后的副产物

在普通向量 memory 里，关系通常要等召回后再靠模型推。  
在 `Graphiti` 里，关系本身就是存储对象。

这对多跳问题尤其关键。  
很多系统在“单条记忆召回”上没问题，但一到“谁依赖谁”“哪个事实关联哪个事件”就开始崩。  
Graphiti 在工程上最值得学的，就是它把这类问题提前结构化了。

### 2. 时间有效性要做成显式结构

这是它比一般图记忆项目更有价值的地方。  
很多系统会把时间当 metadata 存一下就算了，但真正难的是：

- 事实从什么时候开始成立；
- 到什么时候不再成立；
- 新事实出现时旧边怎么处理；
- 历史问题和当前问题能不能用不同视角回答。

这类设计会直接决定系统能不能真正处理时间型长期记忆。

### 3. 原始 episode 和图事实都要保留

如果一个图系统只剩结构，没有回到原始交互的能力，后面很容易出现“图是对的但语义不够”的问题。  
你读 Graphiti 时，要特别留意它如何处理原始来源与图内结构的连接。

### 4. 图系统的难点不在建图，而在持续更新

图 memory 最难的不是第一次抽出三元组，而是：

- 第二次相似信息进来怎么办；
- 新旧关系冲突怎么办；
- 一次临时状态要不要覆盖长期结构；
- 检索时如何避免图变成噪声迷宫。

读 Graphiti 的正确心态，不是“它把图建出来了”，而是“它怎样把图维护下去”。

### 5. 它补的是向量 memory 的结构短板

如果 `Mem0` 是“面向产品化的摘要型 memory layer”，那 `Graphiti` 更像是对这一路线的结构增强版本。  
你应该把它放在脑中与 `Mem0` 对照着看，而不是把两者想成竞争替代品。

## 五、不要误读 Graphiti 的地方

第一，不要把它看成“普通知识图谱项目”。  
它不是静态企业知识图谱，而是为 agent memory 设计的动态 temporal graph。

第二，也不要以为图结构天然就更强。  
图带来的代价是维护复杂度。你只有在关系、依赖、时间有效性真的是核心问题时，Graphiti 这类结构才会显著值回票价。

第三，不要拿它学“完整 agent runtime”。  
那一块 Letta 更合适。Graphiti 更适合学图 memory engine 本身。

## 六、最适合谁读

`Graphiti` 最适合：

1. 做 CRM / support / workflow dependency 系统的人
2. 想研究时间知识图 memory 的人
3. 想补足纯向量 memory 结构短板的人

## 七、最后一句总结

如果让我一句话定义 `Graphiti` 的学习价值，我会写：

> **它展示了长期记忆怎样从“语义相似度仓库”升级为“带时间和关系的可演化事实图”。**

这就是它最值得认真研究的地方。
