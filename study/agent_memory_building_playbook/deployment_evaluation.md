# 部署与评测：让 memory system 真的长成一个系统

很多 memory 项目在 demo 阶段看起来都不错，但一到真实使用就会暴露问题。  
原因通常不是“模型突然不行了”，而是工程层没补上：

- 没有审计
- 没有回归评测
- 没有部署边界
- 没有多用户隔离
- 没有成本监控

## 一、部署时最少要想清楚什么

### 1. 多用户 / 多 workspace 隔离

长期记忆如果串用户，是致命事故。

### 2. 索引刷新策略

memory 更新后，向量索引和图索引多久同步。

### 3. 成本监控

写入、更新、检索各自耗多少 token。

### 4. 审计日志

要能查：

- 这条 memory 怎么来的
- 何时被改过
- 谁在回答时用过它

## 二、评测最少要保留哪几类题

1. 单跳事实题
2. 更新覆盖题
3. 多跳组合题
4. 时间推理题
5. 冲突与拒答题

## 三、一句话总结

真正的 memory system 不是“能写、能查”就结束了，而是要能被部署、被监控、被回归测试、被审计。  
只有这样，它才从能力 demo 变成工程系统。
