# 核心点一：信息提取（Information Extraction）

信息提取是整个 Agent Memory 生命周期的第一步，也是最容易被低估的一步。很多人会觉得 memory system 的关键在“怎么存”和“怎么查”，但如果一开始写进去的东西就不对，后面的管理、存储和检索都只是把错误放大。  
从工程角度看，信息提取实际上是在回答三个问题：

1. 这条交互里什么值得写入长期记忆？
2. 这条信息应该以什么粒度进入记忆系统？
3. 写入时保留原话、压成摘要，还是转换成结构化关系？

统一框架把这一层大致归成三种范式：`直接归档`、`摘要式提取`、`图式提取`。这三个范式不是互斥关系，很多强方法其实都是 hybrid。差别不在于“有没有用 LLM”，而在于**写入后的信息密度、可还原性和可维护性**。

## 一、直接归档：保留原始消息，优点是信息完整

直接归档是最朴素的写法：把用户消息、系统回复、时间戳、会话元数据直接落库，然后把“到底哪条重要”这个判断延后到检索阶段。  
代表方法是 `MemoryBank` 和 `MemGPT`。它们的共同点是：原始语言表述没有在写入时被强压缩，所以像语气、限定条件、上下文位置这些细节仍在。

这类方法的优点很明显：

- 不容易在写入期丢语义；
- 后面遇到复杂问题时，仍然可以回看原话；
- 如果系统未来要做更好的摘要器或更好的检索器，历史原始数据还能重新加工。

但代价也很直接：

- 冗余极高；
- 时间一长以后，相似信息会越积越多；
- 真正的“什么重要”被推迟到了回答阶段，导致检索负担变重；
- 如果没有良好的更新和关联机制，最终容易演化成“可召回的聊天记录仓库”，而不是“可操作的长期记忆系统”。

因此，直接归档更像是一个**保守但厚重的底座**。它不是没价值，恰恰相反，它在高风险场景里很重要，因为你总能回到原始证据。

## 二、摘要式提取：把信息压缩成更高密度的记忆单元

摘要式提取的核心目标，是把一段自然语言交互压成更便于长期管理的单位。例如：

- 用户偏好摘要；
- 稳定事实摘要；
- 任务状态摘要；
- 关键词、标签、上下文描述；
- 面向后续检索的规范化 note。

`A-MEM`、`Mem0`、`MemOS` 都属于这一大类，只是强度不同。`A-MEM` 更强调 note 化和链接化，`Mem0` 更强调生产可用的 salience extraction 与更新，`MemOS` 则把摘要写入纳入更大的 memory OS 视角。

摘要式提取为什么流行？因为它解决了直接归档最致命的两个问题：

1. 原始会话太长，不能都带着走。
2. 很多回答场景并不需要整段原文，只需要“抽出来的稳定事实”。

但这类方法也引入了一个新的核心风险：**摘要漂移**。  
一旦摘要器把一句话概括错了，错误就会以一种“更高级、更稳定、更像事实”的形式进入长期记忆，比原始记录里的错误更难被发现。  
比如用户说：“最近可以少量喝牛奶了，但点咖啡还是优先燕麦奶。”  
如果摘要器只写成“用户现在可以喝牛奶”，那后面整个推荐行为都会被带偏。

所以摘要式提取的真正难点，不是“能不能压缩”，而是：

- 能不能保住限定条件；
- 能不能区分长期事实和短期变化；
- 能不能在必要时回到原始证据；
- 能不能和后续更新机制协同，而不是每次都重写成全新摘要。

## 三、图式提取：把记忆写成实体、关系和时间结构

图式提取比摘要式再进一步。它不只是“压缩文本”，而是把文本映射为实体、关系、属性和时间元数据，形成可以做图遍历的结构。  
`Mem0g` 和 `Zep / Graphiti` 是最典型的代表。

这类方法的魅力在于：很多现实问题天然就是关系问题，而不是单句摘要问题。例如：

- “这个联系人属于哪个账户？”
- “这个项目上线前依赖哪两个审批？”
- “这个偏好是从什么时候开始成立的？”

如果这些信息只保存在摘要文本里，后续需要靠向量相似度和 LLM 推理来拼；但如果提取时就写成结构化边，后续查询会更直接。

不过图式提取最常见的问题也很明显：**结构越清晰，细语义越容易丢。**  
三元组很适合表达“谁和谁有什么关系”，但不擅长表达：

- 语气强弱；
- 例外条件；
- 条件范围；
- 微妙的时间切换；
- 模糊但重要的叙述性背景。

这也是为什么统一框架里 `Mem0` 在很多场景下优于 `Mem0g`：图式提取虽然带来了更好的结构检索潜力，但也更容易在写入时损失原始意义。

## 四、为什么最强方法几乎都不是单一提取范式

从论文和大仓库实现一起看，一个很清楚的趋势是：**成熟系统越来越少用单一提取范式。**  
真正强的系统通常是组合拳：

- 原始消息保留，用来做审计和证据回溯；
- 摘要记忆，用来降低长期维护成本；
- 结构化关系，用来支持多跳和依赖检索。

这就是为什么 `Mem0`、`MemoryOS`、`Graphiti` 这类工程上更强的项目，看起来都不像“只有一种 memory object”。  
它们往往会同时维护：

- raw episode；
- summary memory；
- relation / entity layer；
- metadata，例如时间、来源、重要性、版本号。

从工程角度讲，这种设计的好处是：不同 downstream 任务可以选择不同粒度的信息。  
简单偏好问答走摘要层；复杂争议问题回原始层；关系依赖问题走结构层。

## 五、如果自己设计系统，这一层怎么落地

如果你要做自己的 memory agent，我会建议把信息提取这一层设计成“分两段”：

第一段是 **保守写入**：

- 原始交互先落原文；
- 记录时间、会话、参与者、来源；
- 不在这一层做激进裁剪。

第二段是 **派生写入**：

- 用专门的 extractor 生成摘要事实；
- 对稳定偏好、长期事实、项目状态生成结构化 note；
- 对明确关系类信息再额外写图边或 schema object。

这样做的价值在于：  
你不会把“第一层是否存下来”和“第二层怎么抽象”绑死。后面如果摘要器升级，或者关系抽取规则变化，原始数据仍然能重跑。

## 六、这一层的设计原则

最后，把信息提取浓缩成几条可操作原则：

1. **先保住原始证据，再做压缩。**
2. **摘要不要试图替代原文，只应该补充原文。**
3. **关系提取适合增强检索，不适合单独承担全部语义。**
4. **写入时必须区分稳定信息和临时信息。**
5. **提取层要为后续更新层服务，而不是孤立地追求“摘要好看”。**

所以，信息提取层真正解决的问题不是“怎么把聊天存起来”，而是：  
**怎么把未来真正值得反复使用的信息，以一个既不会太稀、也不会太粗的形式写进系统。**
