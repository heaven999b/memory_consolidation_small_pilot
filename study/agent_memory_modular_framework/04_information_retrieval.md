# 核心点四：信息检索（Information Retrieval）

信息检索是 memory system 对外显现能力最直接的一层。  
无论前面提取、管理、存储做得多复杂，用户最后感受到的往往还是一句话：**系统到底能不能在需要的时候把该想起来的东西想起来。**

统一框架把检索范式拆成四类：

1. 词汇检索
2. 向量检索
3. 结构检索
4. LLM 辅助检索

这四类方法并不是互斥关系。真正成熟的系统，通常都会把它们组合起来，而不是单押某一种。

## 一、词汇检索：最朴素，但在很多场景依然必要

词汇检索基于表层 token 或术语重叠，例如 BM25、Jaccard 或 keyword match。  
它经常被低估，因为大家更容易把注意力放在 embedding 上。但在长期记忆场景里，词汇检索依然有明确价值：

- 专有名词精确命中；
- 项目代号、账户名、人名、工单号；
- 时间表达、版本号、错误码；
- 用户自己定义的标签或关键词。

这类检索的优势是高精度、低解释成本。  
缺点则是明显的语义脆弱性：一旦用户换了说法、用了代称、做了改写，单纯词汇匹配就很容易失灵。

所以词汇检索通常不应该单独承担整个 memory retrieval，而更适合做：

- 第一层精确召回；
- 向量检索的补充通道；
- rerank 特征的一部分。

## 二、向量检索：当前最主流，但不是万能药

向量检索是今天最普遍的 memory retrieval 方式。  
它把记忆单元编码到连续向量空间，通过语义相似度召回 top-k 候选。`Mem0`、`MemoryOS`、`Letta` 等大部分完整仓库都把它作为基础能力。

它流行的原因很简单：

- 能解决表述变化问题；
- 对自然语言非常友好；
- 与现有 embedding 和 ANN 生态兼容；
- 很适合从摘要或 note 里召回候选上下文。

但向量检索在长期记忆里有两个典型问题。

第一个问题是 **相似不等于相关**。  
语义相似度高的内容，不一定是当前任务真正需要的证据。它可能只是主题接近，却不包含你要的时间点、状态变化或例外条件。

第二个问题是 **结构消失**。  
在纯向量空间里，“这个 deadline 是更新后的版本”与“这条审批是上线前置依赖”这样的结构关系并不显式存在，往往还得靠 LLM 在读出来之后自己再推。

所以向量检索是主力，但它通常只适合做“候选召回层”，而不是整个问题解决层。

## 三、结构检索：多跳和依赖问题的核心武器

结构检索依赖图、树、层级存储或显式链接来找信息。  
`Mem0g`、`Zep / Graphiti`、`MemTree`、`MemOS` 这一类方法都大量利用结构检索，只是结构形态不同。

结构检索的关键优势在于：它把“关系”从隐式推理对象变成显式导航对象。  
当用户问：

- “这个项目还缺什么前置条件？”
- “这个联系人和哪个账户、哪个事件有关？”
- “这个偏好是什么时候开始成立的？”

这类问题如果只走向量召回，系统往往需要：

1. 先找几条看起来相关的文本；
2. 再让 LLM 在文本里找关系；
3. 最后再做多跳组合。

而结构检索可以直接从边和节点出发，把问题变成：

- 查节点；
- 顺边遍历；
- 在时间窗口内筛；
- 必要时再回源。

这就是为什么 `Graphiti` 这种项目，会把 temporal validity、episode provenance、entity summary 都设计成结构层的一部分。它不是为了炫技，而是因为很多真实问题就是图问题。

## 四、LLM 辅助检索：让模型参与“怎么找”，而不只是“怎么答”

LLM 辅助检索的思路，是让模型在检索前或检索中发挥作用，例如：

- 改写 query；
- 抽取关键实体；
- 识别时间约束；
- 决定应该查哪个 memory tier；
- 根据前一轮召回结果继续追问或扩展搜索。

`MemoChat` 比较强调这一侧，`MemGPT` 的 agentic 检索过程也带这种味道。  
这类方法的好处，是能处理模糊查询和复杂任务。用户的问题往往不是“请按数据库键值精确查询”，而是“你还记得我们上个月说过那个计划吗？”

LLM 辅助检索把模型从一个被动 reader 变成主动检索 orchestrator。  
但代价也很明显：

- 成本高；
- 行为不稳定；
- 调不好时会过度检索；
- 一旦前面候选召回就错了，LLM 可能只是在错候选上做更复杂的推理。

因此，成熟系统里 LLM 辅助检索更适合放在：

- query parsing；
- retrieval routing；
- candidate reranking；
- ambiguous query clarification；

而不是把每次检索都完全交给模型自由发挥。

## 五、为什么检索层不能单独看

统一框架里非常重要的一点是：**检索效果不是只由检索器决定的。**  
如果前面的信息提取过粗、记忆管理混乱、存储结构设计不合理，那么检索层无论多先进，都只能在低质量记忆中挑较不坏的结果。

这也是为什么很多工程项目里，真正有效的改进不是“换一个更强 embedding”，而是：

- 改善写入粒度；
- 给长期事实加时间标签；
- 把冲突事实分版本保存；
- 引入层级或结构关系。

一旦前面这些做好，检索本身反而可以更简单、更稳、更便宜。

## 六、工业级系统更像“混合检索管线”

如果你去看 `Mem0`、`Letta`、`Graphiti`、`MemoryOS` 这种完整项目，会发现它们在 spirit 上几乎都不是单一路径检索，而更像混合管线：

1. 先根据 query 类型判断该走哪一层 memory。
2. 同时做词汇或向量召回。
3. 如有结构关系，再补图遍历或层级扩展。
4. 取回候选后，再由模型做读、比对、选择和回答。

这意味着真正成熟的 retrieval system，关注的不是“哪种检索范式唯一正确”，而是：

- 什么问题先走哪种召回；
- 候选集怎么合并；
- 什么情况下需要回源；
- 什么情况下应该拒答或追问。

## 七、自己做项目时的一个实用配置

如果你在做一个长期交互 agent，我会建议先用下面这个检索顺序：

1. **先做 query parsing**
   把问题拆成实体、时间、任务意图、是否需要多跳。

2. **再做混合召回**
   词汇检索召回显式实体，向量检索召回语义候选。

3. **如有结构层，再做图或层级扩展**
   例如 deadline 相关问题就顺着项目节点往前置依赖走。

4. **最后做 raw fallback**
   如果候选之间存在冲突，或者当前问题对精确性要求很高，就回到原始消息、原始 note、原始 episode 去校验。

这套流程的价值在于：它把“查找”变成一个可控管线，而不是单次 top-k 抽奖。

## 八、这一层的结论

检索层不是一个孤立模块，而是整个 memory system 的对外接口。  
一个好的检索层不只是“召回得多”，更重要的是：

- 召回得准；
- 召回得可解释；
- 能处理结构关系；
- 在冲突和不确定场景下知道何时回源。

所以最终最值得记住的一句话是：

> **向量检索解决“像不像”，结构检索解决“连不连”，LLM 辅助检索解决“该怎么找”。**

真正强的系统，往往不是三选一，而是把三者排成顺序、各司其职。
