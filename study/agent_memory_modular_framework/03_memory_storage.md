# 核心点三：记忆存储（Memory Storage）

很多人在谈 Agent Memory 时，第一反应是“存在哪个向量库里”。这其实把问题想得太窄了。  
记忆存储真正要回答的，不只是“放在哪”，而是：

1. 这些记忆应该如何被组织？
2. 记忆单元的表示形式是什么？
3. 原始证据、抽象摘要、关系结构之间怎么共存？

统一框架从两个维度来拆存储层：`组织方式` 和 `表示方式`。前者讲层次结构，后者讲数据形态。

## 一、组织方式：平层存储 vs 分层存储

平层存储最容易理解：所有 memory item 都放在一个统一池子里，不区分短期、中期、长期，也不区分概念层和细节层。  
这类设计实现简单，读写直接，前期上手快。`MemoryBank` 和较早期的 `Mem0` 视角，都可以大致归到这一侧。

平层存储的优势是：

- 结构简单；
- 很适合快速实验；
- 数据模型容易统一；
- 检索入口单一，便于开发。

但它的代价会随着规模增长越来越明显：

- 临时消息和长期事实互相污染；
- 相似记录会在一个池子里竞争；
- 越新的内容越容易压住越老但更稳定的内容；
- 后期为了不漏召回，只能提高检索成本。

分层存储就是为了解决这些问题。  
`MemGPT` 的 memory tier、`MemoryOS` 的短中长期、`MemOS` 的更广义 memory hierarchy，本质上都是在做同一件事：**把不同时间尺度、不同抽象粒度的信息分开管理。**

分层的价值在于：

- 让近期上下文保持敏捷；
- 让中期项目状态有独立空间；
- 让长期偏好和长期事实不被日常噪声稀释。

从工程角度看，分层存储不只是“多建几个表”，而是让系统拥有不同的 retention policy、update policy 和 retrieval policy。

## 二、表示方式：向量、图、树，各自擅长什么

表示方式决定的是 memory object 本身长什么样。

### 1. 向量表示

这是最主流的一类。  
文本、摘要、事实描述先被编码为 embedding，再通过余弦相似度或 ANN 索引做检索。`Mem0`、`MemoryOS`、`Letta` 这一大类项目都 heavily 依赖向量表示。

向量表示的优势是通用：

- 对自然语言兼容性强；
- 不要求开发者提前定义结构；
- 现成基础设施丰富；
- 与现代 RAG 生态衔接自然。

但它的天然短板也很清楚：

- 关系结构不显式；
- 对时间演化不敏感；
- “为什么相关”常常不透明；
- 复杂多跳问题需要额外层来补。

### 2. 图表示

图表示把实体、关系和时间窗口显式化，典型代表是 `Zep / Graphiti`、`Mem0g`。  
图的好处，是能天然支持：

- 谁和谁相关；
- 哪些条件是依赖关系；
- 哪个事实在什么时候成立；
- 哪条边来源于哪段 episode。

这类表示特别适合 CRM、support、project dependency、temporal facts 这些场景。  
但它也要求系统承担更多维护成本：

- 图 schema 怎么定；
- 去重怎么做；
- 同一实体的多次演化怎么表达；
- 图更新和图查询怎样保持性能。

所以图表示不是“更先进的向量”，而是另一种更强结构、也更重维护的选择。

### 3. 树表示

树表示最典型的是 `MemTree`，以及在更宽泛意义上的 `MemOS`。  
它的特点不是强调实体关系，而是强调抽象层级：

- 叶子节点保留细粒度信息；
- 上层节点逐步聚合成更高层 schema 或摘要；
- 检索时既可以从上往下缩小范围，也可以回到底层查证据。

树表示最大的价值在于：它把“压缩”和“保真”统一到了一个结构里。  
你不需要在“只保留摘要”和“只保留原文”之间二选一，而是让不同层保留不同抽象度。

## 三、为什么树和分层方法在论文里整体更强

统一框架的实验结论里，一个非常稳定的信号是：**树状或分层方法整体更强。**  
这不是偶然，也不是因为它们“看起来更复杂”，而是因为它们更符合长期记忆问题本身的结构。

长期交互里天然存在两类张力：

1. 你想保留细节，否则回答会失真。
2. 你又不能把所有细节都直接带进上下文，否则成本和噪声都会爆炸。

树和分层系统之所以强，是因为它们不强迫你二选一。  
它们允许系统同时拥有：

- 可回溯的底层证据；
- 可快速访问的中层摘要；
- 稳定、抽象、长期的高层概念。

这也是 `MemTree`、`MemoryOS`、`MemOS` 会在统一框架里整体表现亮眼的深层原因。

## 四、工业实现里，存储层还要考虑什么

论文通常把存储讲成“平层/分层、向量/图”就差不多了，但真正的工业实现还必须考虑更多问题：

- **可审计性**：原始 memory object 能不能追溯到来源？
- **版本控制**：一个摘要被更新后，旧版本还在不在？
- **冷热分层**：高频使用和低频归档是不是需要不同存储策略？
- **索引重建**：摘要或图边更新后，索引如何同步？
- **多租户隔离**：不同用户、不同 workspace、不同 agent 的 memory 是否彻底隔离？

你如果去看 `Letta`、`Mem0`、`Graphiti`、`MemoryOS` 这类完整仓库，会发现它们的复杂度往往不在某一个 fancy 算法上，而是在这些“脏活”上。

比如：

- `Letta` 要处理 agent state、DB schema、runtime persistence；
- `Mem0` 要同时处理 API、server、integration、evaluation；
- `Graphiti` 要处理 provenance、temporal validity、graph core 与 server 层；
- `MemoryOS` 要把分层 memory 落到 package、MCP、playground 和 eval 里。

这说明存储层不是一张表或者一个向量库配置，而是**整个系统的数据骨架**。

## 五、怎么为自己的项目选存储形态

如果你在做个人助理、工作台、日常知识助手：

- 优先选分层 + 向量；
- 原始消息单独留存；
- 长期稳定偏好和项目事实独立存。

如果你在做 coding agent：

- 更适合“树式摘要 + raw evidence fallback”；
- 因为代码问题经常需要抽象理解，但又必须能回 diff、log、test 输出。

如果你在做 CRM、支持系统、流程依赖系统：

- 图层会比纯向量更值钱；
- 因为“谁和谁有关”“哪件事依赖哪件事”是第一公民。

如果你还不确定需求：

- 不要一上来就做全图或全树；
- 先做 raw + summary + tiered vector；
- 等多跳关系问题真的成瓶颈，再加图或树。

## 六、存储层的核心结论

记忆存储层不是一个“基础设施选择题”，而是一道“信息组织哲学题”。  
你选择的不是 Chroma、Qdrant 或 Neo4j 本身，而是：

- 你的系统把什么视为稳定知识；
- 你的系统是否允许不同粒度的信息共存；
- 你的系统是否给未来更新和检索留下了空间。

所以一个好的 memory storage 设计，应该同时做到三件事：

1. **保住底层证据**
2. **支持上层抽象**
3. **为更新与检索留出结构余地**

当这三件事被同时满足时，存储层才真正配得上“长期记忆系统”的名字。
