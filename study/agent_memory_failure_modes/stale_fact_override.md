# 失败模式二：旧事实覆盖失败（Stale Fact Override Failure）

旧事实覆盖失败指的是：系统已经接收到更新后的新事实，但旧事实仍然以高权重残留在 memory 里，最终在回答时和新事实竞争，甚至压过新事实。

这是长期 memory agent 最现实的失败之一，因为真实世界里的很多信息都不是静态的：

- deadline 会改
- 联系方式会变
- 偏好会漂移
- 负责人会换
- 项目状态会推进

如果系统只会“累积记忆”，不会“有区分地覆盖记忆”，那它很快就会变成一个新旧事实混杂的系统。

## 一、这个失败通常长什么样

典型场景：

1. `S1`：“项目 deadline 是 7 月 15 日。”
2. `S4`：“deadline 改成 7 月 22 日。”
3. 用户后来问：“现在最终 deadline 是多少？”

失败系统的行为通常有三类：

- 直接回答旧值 `7 月 15 日`
- 把两个值都吐出来，让用户自己判断
- 说“之前是 7 月 15 日，后来改了”，但最终默认仍引用旧值

这三类其实本质一样：**系统没有真正把新事实提升为当前主事实。**

## 二、为什么这个失败很常见

因为旧事实覆盖失败不是单一模块的问题，而是跨层问题：

- 提取层没识别“这是更新，不是新条目”
- 管理层没把新旧事实绑定
- 存储层没保留版本关系
- 检索层把旧事实和新事实同时召回
- 回答层没能力判断哪个当前有效

所以你不能只怪 retrieval，也不能只怪 summarizer。  
它往往是整条 memory lifecycle 协同失败。

## 三、哪些信息最容易出现覆盖失败

### 1. 时间性强的项目状态

- deadline
- milestone
- 当前 owner
- 当前 blocker

### 2. 用户偏好变化

- “以前喜欢 A，现在更偏 B”
- “目前先不要推荐 C”

### 3. 联系方式和身份属性

- 手机号
- 邮箱
- 所属团队
- 当前职位

### 4. 会周期性变化的业务状态

- 预算
- 合同进度
- 客户风险等级

## 四、哪些方法容易在这里翻车

平层原文保留但缺少显式更新绑定的方法最容易出问题，例如：

- `MemoryBank`
- 一些轻量 raw archive 系统
- 只有向量召回但没有版本语义的实现

而 `Mem0`、`MemoryOS`、`Graphiti` 这类如果实现得好，理论上更有机会处理好，因为它们更强调：

- 相似项更新
- 层级迁移
- 时间或关系结构

但“理论上能处理”不等于“默认就处理好”，这取决于具体更新规则和版本语义是否做实。

## 五、怎么发现这个问题

### 1. 看 recall candidate 里是否同时充满新旧版本

如果一个问题的候选集里总是把旧事实和新事实混在前几位，而且没有显式 current flag，这是危险信号。

### 2. 看 memory object 有没有“当前有效”语义

如果系统只有文本摘要，没有：

- version
- valid_from
- valid_to
- supersedes
- current

这类字段，那么覆盖失败几乎迟早会发生。

### 3. 看系统是否把更新理解成 append

如果新消息一律 append 成新条目，而没有判断“这是在改旧事实”，那长期下来会不断制造冲突记忆。

## 六、怎么缓解

### 1. 建立显式版本关系

每条会变化的事实最好都支持：

- 当前版本
- 历史版本
- 被谁覆盖
- 覆盖时间

### 2. 更新时先做 entity / field 对齐

在把新信息写入前，先判断：

- 它在说谁
- 它在改哪个字段
- 它是新增属性还是更新属性

### 3. 检索阶段优先 current state

对明确是“当前值”的问题，检索器不应把所有历史版本一视同仁，而应该先优先 current 或最新有效对象。

### 4. 保留历史，但不要让历史和当前同权竞争

很多系统怕丢信息，于是什么都留着。留着没问题，问题在于不能让历史版本在回答“当前状态”时和当前版本平权竞争。

## 七、如果自己设计系统，最稳的做法是什么

我建议至少把会变化的 memory object 分成两层：

1. **历史事件层**
   保存“发生了哪些变更”

2. **当前状态层**
   保存“现在有效的是什么”

这样做的好处是：

- 你不会丢历史
- 你也不会每次都在所有历史里重新推当前值

这在项目状态、用户画像、CRM 系统里都特别有用。

## 八、最后一句总结

旧事实覆盖失败的根源不是系统记得太少，而是系统**不会区分“曾经真实”和“现在真实”**。  
长期 memory 一旦进入真实业务，这个问题几乎一定会出现，所以必须被显式设计，而不能寄希望于回答时临场推理补救。
