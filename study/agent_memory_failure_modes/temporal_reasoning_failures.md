# 失败模式四：时间推理失败（Temporal Reasoning Failures）

时间推理失败是长期 memory 系统最容易被低估、也最难被彻底补好的问题之一。  
它不是简单的“日期没记住”，而是系统在处理“什么时候成立、什么时候失效、哪件事先发生、哪个状态当前有效”这些问题时出错。

统一框架里一个很重要的发现是：很多方法在时间任务上的进步，部分来自 backbone LLM 变强，而不完全来自 memory 架构本身。  
这恰恰说明：**时间处理仍然是现有系统的短板。**

## 一、时间失败通常长什么样

典型情况有：

1. 旧事实和新事实混淆
2. 把“曾经成立”当成“现在成立”
3. 无法判断事件先后
4. 知道几个事实，却不知道它们在时间线上怎么排

例如：

- “上个月负责这个项目的是 Alice，这个月换成 Bob。”

失败系统可能会：

- 回答 Alice
- 回答 Bob
- 说 Alice 和 Bob 都负责过，但答不出“现在是谁”

这种错误本质上不是 recall 错，而是 temporal state resolution 错。

## 二、为什么时间问题比普通事实问题难

因为时间不是一个普通字段，它会影响：

- 哪条 memory 现在有效
- 哪条 memory 只是历史记录
- 哪个关系应该被优先使用
- 哪个冲突其实不是冲突，而是时间切换

换句话说，时间不是附属 metadata，而是决定语义解释方式的主条件。

## 三、哪些场景最容易暴露时间失败

### 1. 项目管理场景

- deadline 更新
- owner 变更
- blocker 清除与新增

### 2. 用户偏好漂移

- 以前喜欢 A，现在喜欢 B
- 最近临时改变，但长期偏好未变

### 3. CRM / support / workflow

- 工单状态推进
- 风险等级调整
- 联系人角色变动

## 四、哪些系统更容易出现时间失败

所有“只会存文本，但不显式建模时间有效性”的系统都容易出现这个问题。  
区别只是：

- 有些系统错得明显；
- 有些系统靠更强模型暂时兜住一部分。

平层摘要系统尤其危险，因为时间变化很容易在多轮摘要里被洗平。  
图系统如果没有 temporal edge 或 validity window，也一样会出问题。  
所以不是“用图就自动解决时间”，关键还是是否显式建模时间。

## 五、怎么发现系统在这里有问题

### 1. 同类问题换个时间问法就不稳

比如：

- “现在是谁负责？”
- “之前是谁负责？”
- “是先改 deadline 还是先换负责人？”

如果系统对这三种问法稳定性差异很大，那就是时间层有漏洞。

### 2. 回答总是倾向最近看到的内容

这说明系统有明显 recent bias，但没有真正的 temporal resolution。

### 3. 中间 memory object 缺少时间有效性字段

如果 memory 只有文本和 embedding，没有：

- valid_from
- valid_to
- timestamp role
- event order
- current state flag

那么时间失败只是迟早问题。

## 六、怎么缓解

### 1. 把时间做成显式结构

不要只是把时间词留在文本里，而要有显式字段或关系。

### 2. 区分事件记录和当前状态

历史事件是一层，当前状态是一层。  
这比每次从所有历史里重新推当前值稳得多。

### 3. 检索时带时间约束

对“现在”“之前”“当时”“后来”这类 query，检索逻辑应该显式处理 temporal filter，而不是只做普通相似度召回。

### 4. 回答阶段允许回源

如果时间冲突明显，系统最好能回原始事件序列，而不是强行猜。

## 七、自己做系统时最值得加的两样东西

1. **状态层 + 事件层双表示**
2. **time-aware retrieval filter**

前者保证你不会只剩历史流水；后者保证你问“当前是什么”时不会把一堆旧事实都平等召回。

## 八、最后一句总结

时间推理失败不是“系统不聪明”，而是**系统没有把时间当成记忆语义的一部分来建模**。  
只要你的项目里事实会变、状态会推进、偏好会漂移，时间问题就一定会成为核心问题，而不是边角问题。
