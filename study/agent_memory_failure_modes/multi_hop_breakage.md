# 失败模式三：多跳检索断链（Multi-hop Retrieval Breakage）

多跳检索断链指的是：系统明明分别记住了几个局部事实，但在需要把它们串起来时失败了。  
也就是说，记忆不是“没有”，而是**记忆之间没有形成可用路径**。

这是为什么很多系统单跳看起来不错，但一到复杂问题就明显掉分的核心原因。

## 一、典型表现

例如系统分别记住了：

- “Atlas 项目负责人是 Bob”
- “上线前必须做安全评审”
- “安全评审负责人是 Carol”

用户问：

> “Atlas 项目上线前还需要谁参与？”

一个单跳系统会觉得每条都相关，但它未必能稳定地走出这条链：

`Atlas -> 上线前依赖安全评审 -> 安全评审负责人 Carol`

于是常见失败会是：

- 只答 Bob
- 只答需要安全评审，但不答 Carol
- 回答一堆相关片段，却不给出真正组合后的结果

## 二、为什么这类失败很常见

因为很多 memory system 其实是“候选召回系统”，不是“关系连接系统”。  
它们能找回相关片段，但不能保证片段之间已经存在可操作的连接结构。

造成断链的常见原因有：

1. memory item 之间没有显式关系
2. 关系存在，但检索阶段不会沿关系扩展
3. 时间或版本把链条打断了
4. 摘要把连接点洗掉了

## 三、哪些问题最容易触发断链

### 1. 依赖型问题

- 上线前还缺什么
- 这个工单依赖哪个审批
- 哪个项目受哪个 blocker 影响

### 2. 人-事-物三元关系问题

- 谁负责哪个项目的哪个环节
- 哪个联系人关联哪个客户和哪个事件

### 3. 时间路径问题

- 这个变化从什么时候开始
- 旧状态怎么变成现在的

## 四、哪些系统更容易断链

纯平层、纯向量、弱结构系统最容易断：

- 原始消息保留但不显式连边
- 摘要池里缺少 cross-memory linkage
- retrieval 只做 top-k 相似度，不做结构扩展

这也是为什么统一框架里，`MemoryBank`、`MemGPT`、`MemoChat` 在多跳任务上相对偏弱，而 `A-MEM`、`Zep`、`MemTree`、`Graphiti` 这类方法更占优。

## 五、怎么发现系统在这里出问题

### 1. 单跳没问题，多跳明显掉分

这几乎是最直接信号。  
如果系统能准确答“Bob 负责 Atlas”，也能答“安全评审负责人是 Carol”，但答不好“Atlas 上线前还需要谁参与”，那基本就是典型断链。

### 2. 候选召回很多，但最终答案缺连接

说明 recall 不是主问题，问题在于组合。

### 3. 检索 trace 里没有第二跳动作

如果你把检索过程打出来，会发现系统只做了一次候选召回，没有顺着实体、图边、层级或链接继续扩展。

## 六、怎么缓解

### 1. 引入显式关系

最直接的方法就是：

- 图边
- parent-child 层级
- supersedes / depends_on / belongs_to 这类 schema relation

### 2. 做 retrieval expansion

不是只取 top-k，而是：

1. 先找到第一跳关键对象
2. 再顺着边或层级扩展第二跳候选
3. 必要时再回原始证据

### 3. 保留连接点，不要在摘要里洗掉

如果摘要把“依赖谁”“属于谁”“因为什么变化”这类关系性词语压掉，后面就算想扩展也没起点。

## 七、自己做系统时该怎么设计

如果你明确知道自己的场景需要多跳，至少要有其中一种：

1. 显式图结构
2. 显式 memory links
3. 层级树结构
4. 基于 schema 的依赖关系字段

如果四种都没有，那你本质上是在指望 LLM 每次回答时临场把若干相似片段重新拼起来，这在大规模系统里会非常不稳。

## 八、最后一句总结

多跳检索断链不是“没记住”，而是**记忆没有被组织成网络**。  
所以解决它的关键，不是把 embedding 再调好一点，而是让 memory objects 之间真的形成可走通的路径。
